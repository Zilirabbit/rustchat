# RustChat 需求分析文档

## 1. 文档目的

本文档用于从需求分析角度整理 RustChat 项目的当前功能范围、阶段演进、用户场景、业务规则、范围边界与验收标准。

本文档重点回答以下问题：

- RustChat 为什么要做
- RustChat 当前要解决哪些核心用户问题
- 项目已经具备哪些能力
- 各阶段需求如何逐步演进
- 当前版本做到什么程度可以认为满足要求

本文档不替代已有接口文档和设计文档。接口请求、响应字段、错误码与调试示例以 `docs/api/http-api.md` 为准；数据库结构、模块设计与实现细节以 `docs/architecture/db.md`、`docs/design/`、`docs/summary/` 及代码实现为准。

## 2. 项目背景与目标

RustChat 是一个基于 Rust 后端与 Web 前端实现的轻量级即时聊天系统。项目定位是课程作业、练手项目和软件工程流程实践项目，目标是完整走通一个 Web IM 系统从需求、设计、开发、联调到验收的基本流程。

RustChat 不追求商业级 IM 的完整能力，而是围绕“基础聊天闭环”进行需求收敛。当前项目更关注以下目标：

- 实现可运行的 Web 聊天系统
- 支持用户注册、登录与身份认证
- 支持私聊和群聊基础通信
- 支持 WebSocket 实时消息收发
- 支持消息与会话数据持久化
- 支持历史消息、会话列表和未读状态展示
- 支持基础文件消息能力
- 提供可演示的前端聊天界面
- 通过文档、phase 总结和验收记录体现软件工程过程

因此，RustChat 的需求分析应突出“核心场景是否闭环、业务规则是否清晰、验收路径是否可执行”，而不是追求复杂社交、分布式、高并发或商业运营能力。

## 3. 当前项目完成情况

结合当前 API 文档、数据库文档、设计文档、前后端实现说明与 phase 总结，RustChat 已经形成基础可用的聊天系统。

当前已具备的主要能力包括：

- 用户账号能力：支持注册、登录、获取当前用户信息和登录态鉴权。
- 会话能力：支持创建私聊、创建群聊、维护会话成员关系和群成员操作。
- 消息能力：支持文本消息发送、消息持久化、历史消息分页查询和消息实时推送。
- 会话列表能力：支持展示当前用户会话、最近消息、未读数和会话活跃排序。
- 群聊能力：支持群聊创建、添加成员、退出群聊和群消息广播。
- 文件消息能力：支持基础文件上传、文件消息生成与下载访问控制。
- 实时通信能力：支持通过 WebSocket 建立连接、发送消息、接收消息和连接状态维护。
- 前端交互能力：支持登录、聊天页、会话切换、消息展示、消息发送、群成员展示、断线重连和刷新恢复。
- 联调与验证能力：已有 Postman collection、HTTP 联调记录、WebSocket 验收建议、真实数据库测试和 phase 总结文档。

当前项目的需求完成状态可以概括为：用户从登录进入系统，到创建私聊或群聊，再到发送实时消息、刷新后查看历史记录、通过前端完成基础演示，已经形成完整主链路。

## 4. Phase 阶段需求演进

本节是本文档区别于 API 文档和设计文档的重点。RustChat 的需求不是一次性完成的，而是在不同 phase 中逐步收敛和补强。

### Phase 0：项目启动与环境准备

本阶段需求重点是让项目具备基本开发条件。

需求目标：

- 初始化代码仓库和后端工程。
- 明确后端服务可启动。
- 准备前端工程与基础页面结构。
- 建立最小开发环境与文档入口。

需求价值：

- 为后续功能开发提供稳定工程基础。
- 降低环境配置和项目启动成本。

### Phase 1：后端基础设施

本阶段需求重点是建立后端通用能力。

需求目标：

- 提供健康检查和数据库连通性检查。
- 建立统一配置、日志、错误处理和响应结构。
- 接入 PostgreSQL 与 migration 体系。
- 完成核心聊天数据表设计。
- 建立 JWT 鉴权中间件和请求日志能力。

需求价值：

- 让后续用户、会话、消息模块能够复用统一基础能力。
- 确保数据库、鉴权、响应格式和日志排障能力在早期稳定下来。

### Phase 2：实时私聊核心链路

本阶段需求重点是打通最小聊天主链路。

需求目标：

- 支持 WebSocket 连接。
- 支持创建私聊会话。
- 支持通过 WebSocket 发送文本消息。
- 支持消息先持久化再推送。
- 支持会话列表展示。
- 支持历史消息分页查询。

需求价值：

- 首次形成“登录用户 -> 创建会话 -> 发送消息 -> 实时接收 -> 查询历史”的 IM 主链路。
- 为后续群聊、未读状态和前端联调提供可复用基础。

### Phase 3：会话体验与客户端前置接口

本阶段需求重点是补齐前端使用所需的接口和数据一致性规则。

需求目标：

- 增加真实数据库集成测试。
- 支持用户搜索，便于前端创建私聊。
- 支持会话标记已读，便于维护未读状态。
- 增强私聊唯一性，避免同一对用户出现多个私聊会话。

需求价值：

- 从“后端功能可用”推进到“前端可以顺畅接入”。
- 用数据库约束和真实库测试提升业务规则可信度。

### Phase 4：前端 UI 与本地联调

本阶段需求重点是让用户能够通过浏览器完成基础操作。

需求目标：

- 实现基础聊天 UI。
- 完成 HTTP API 本地联调。
- 解决浏览器联调中的 CORS、空会话已读和错误排障问题。
- 支持刷新后恢复当前打开的会话。

需求价值：

- 让项目从“接口可用”转向“用户可操作”。
- 通过真实浏览器场景暴露并修复跨模块问题。

### Phase 5：群聊与一致性补强

本阶段需求重点是扩展多人聊天场景。

需求目标：

- 支持创建群聊。
- 支持添加群成员和退出群聊。
- 支持群消息广播给在线成员。
- 支持群成员列表展示。
- 支持退出群聊后的前端状态清理。

需求价值：

- 将系统从一对一聊天扩展到多人会话。
- 继续复用会话、成员、消息、历史和会话列表能力，验证模块边界的合理性。

### Phase 5.5：WebSocket 体验与消息发送体验优化

本阶段需求重点是改善实时连接和前端使用体验。

需求目标：

- 支持 WebSocket 自动重连。
- 展示在线、连接中、重连中、离线等连接状态。
- 重连成功后重新拉取会话列表和当前会话历史消息。
- 优化消息发送体验，避免离线发送造成状态错觉。

需求价值：

- 提升实时聊天在异常网络或服务重启场景下的可靠性。
- 让用户能清楚理解当前连接状态和消息发送状态。

### Phase 6：真实数据展示与设计资产收口

本阶段需求重点是让前端展示回到真实接口数据。

需求目标：

- 将聊天页从静态 mock 收敛为真实数据最小展示。
- 保留私聊、群聊、成员、WebSocket 文本发送等真实能力。
- 整理样式和设计资产，使页面更适合演示。

需求价值：

- 避免前端展示与后端真实能力脱节。
- 让最终演示更聚焦于真实功能闭环。

## 5. 用户角色与核心场景

### 5.1 用户角色

当前版本只定义一种用户角色：普通用户。

普通用户可以完成以下操作：

- 注册账号
- 登录系统
- 搜索其他用户
- 创建私聊会话
- 创建群聊会话
- 添加群成员
- 退出群聊
- 发送文本消息
- 上传和下载文件
- 查看会话列表
- 查看历史消息
- 接收实时消息

当前版本不区分管理员、审核员、运营人员等复杂角色。

### 5.2 核心场景

#### 场景一：用户注册与登录

用户进入系统后，通过用户名和密码注册账号。注册成功后，用户可以登录并获得访问受保护功能所需的身份凭证。

成功条件：

- 用户可以创建唯一账号。
- 正确账号密码可以登录。
- 登录后可以访问聊天相关功能。

#### 场景二：创建私聊并发送消息

用户搜索另一个用户，创建或复用已有私聊会话。进入会话后，用户发送文本消息；如果对方在线，对方可以实时收到消息；如果刷新页面，双方仍可以通过历史消息看到已发送内容。

成功条件：

- 同一对用户之间不会产生重复私聊会话。
- 消息先保存后推送。
- 在线接收方可以实时收到消息。
- 历史消息能够稳定查询。

#### 场景三：创建群聊并进行多人通信

用户创建群聊并添加成员。群成员进入会话后可以发送消息，在线成员能够实时收到群消息。

成功条件：

- 创建者成为群主。
- 群主可以添加成员。
- 群成员可以接收群消息。
- 非群成员不能访问群聊消息。

#### 场景四：查看会话列表与历史消息

用户登录后进入聊天页，可以看到自己的会话列表。会话列表展示最近消息和未读情况。用户打开会话后，可以加载历史消息，并将当前会话标记为已读。

成功条件：

- 会话列表只展示当前用户相关会话。
- 最近消息和未读数能够反映实际消息状态。
- 历史消息分页查询稳定。

#### 场景五：发送和下载文件消息

用户在会话中上传文件。上传完成后，系统生成文件类型消息，其他有权限的会话成员可以下载。

成功条件：

- 文件上传过程可完成。
- 文件消息能够出现在聊天记录中。
- 非会话成员不能下载对应文件。

#### 场景六：断线重连与刷新恢复

用户在聊天过程中遇到连接异常、后端重启或页面刷新。前端能够展示连接状态，并在恢复连接后重新拉取会话列表和当前会话历史消息。

成功条件：

- 用户可以看到连接状态变化。
- 重连成功后数据能够补偿刷新。
- 页面刷新后可以恢复登录态和当前会话。

## 6. 功能需求分析

### 6.1 用户系统

需求描述：

用户系统负责账号注册、登录认证、当前用户识别和用户搜索，是所有聊天能力的入口。

输入：

- 用户名
- 密码
- 用户搜索关键词
- 登录凭证

输出：

- 用户身份信息
- 登录凭证
- 用户搜索结果
- 鉴权结果

业务规则：

- 用户名必须唯一。
- 密码不得明文存储。
- 受保护功能必须识别当前登录用户。
- 用户搜索只返回创建聊天所需的基础公开信息。

异常情况：

- 用户名已存在时不能重复注册。
- 用户名或密码格式不符合要求时应拒绝。
- 登录凭证缺失、无效或过期时应拒绝访问。
- 搜索不到用户时返回空结果而不是系统错误。

验收标准：

- 新用户可以注册。
- 已注册用户可以登录。
- 登录后可以访问当前用户信息和聊天功能。
- 未登录用户不能访问受保护接口。
- 用户可以通过关键词搜索其他用户。

### 6.2 会话系统

需求描述：

会话系统负责维护私聊和群聊两类会话，以及用户与会话之间的成员关系。

输入：

- 当前用户身份
- 目标用户
- 群聊名称
- 群成员信息
- 会话标识

输出：

- 会话信息
- 成员关系
- 会话已读状态

业务规则：

- 私聊会话由两个用户组成。
- 用户不能和自己创建私聊。
- 同一对用户之间应保持唯一私聊会话。
- 群聊必须有创建者。
- 用户只能操作自己有权限的会话。
- 当前用户进入或处理会话时可以更新已读状态。

异常情况：

- 目标用户不存在时不能创建私聊。
- 非会话成员不能访问会话资源。
- 非群成员不能退出群聊。
- 空会话标记已读应稳定处理。

验收标准：

- 用户可以创建或复用私聊会话。
- 用户可以创建群聊会话。
- 系统可以维护会话成员关系。
- 用户只能看到和操作自己相关的会话。

### 6.3 消息系统

需求描述：

消息系统负责处理消息发送、消息保存、消息查询和消息内容展示。

输入：

- 当前用户身份
- 会话标识
- 消息内容
- 消息类型
- 历史消息分页条件

输出：

- 已保存消息
- 发送确认
- 历史消息列表
- 分页状态

业务规则：

- 只有会话成员才能发送消息。
- 文本消息内容不能为空。
- 消息发送必须先落库，再推送。
- 历史消息查询必须校验成员权限。
- 历史消息应支持分页加载。

异常情况：

- 非成员发送消息应被拒绝。
- 空消息或不合法消息应被拒绝。
- 查询不存在或无权限的会话历史应被拒绝。
- 离线成员不应阻塞发送方获得确认。

验收标准：

- 私聊和群聊中均可发送文本消息。
- 消息刷新后仍可查询。
- 非成员无法发送或查询消息。
- 分页查询结果稳定。

### 6.4 实时通信

需求描述：

实时通信负责用户在线连接、消息实时推送、连接状态维护和异常断线后的恢复体验。

输入：

- 登录凭证
- WebSocket 连接请求
- 客户端消息事件

输出：

- 连接成功事件
- 发送确认事件
- 消息接收事件
- 错误事件
- 连接状态反馈

业务规则：

- WebSocket 连接必须完成身份校验。
- 无效身份不能建立有效连接。
- 发送消息时继续复用会话成员权限校验。
- 前端只在连接可用时发送消息。
- 重连成功后应重新拉取关键数据，补偿断线期间可能漏掉的消息。

异常情况：

- token 无效时连接失败。
- 连接异常断开时前端进入重连状态。
- 重连多次失败后应给用户明确提示。
- 断线期间不应制造“消息已经成功发送”的错觉。

验收标准：

- 合法用户可以建立 WebSocket 连接。
- 用户可以实时收到私聊和群聊消息。
- 连接断开后前端能显示状态变化。
- 重连成功后会话和当前消息能够刷新。

### 6.5 会话列表

需求描述：

会话列表负责展示当前用户参与的所有会话，并提供最近消息、未读数和会话排序信息。

输入：

- 当前用户身份
- 会话和消息状态
- 已读状态

输出：

- 会话列表
- 最近消息摘要
- 未读消息数
- 会话排序结果

业务规则：

- 只展示当前用户所属会话。
- 会话按最近活跃时间排序。
- 未读数按当前用户与会话维度统计。
- 私聊名称应便于用户识别对方。
- 群聊名称应展示群名称。

异常情况：

- 空会话也应能展示或稳定处理。
- 用户退出群聊后，会话列表应移除该会话。
- 重连或刷新后本地会话状态应与服务端对齐。

验收标准：

- 用户登录后可看到自己的会话列表。
- 发送新消息后最近消息和排序更新。
- 打开会话并标记已读后未读数归零。
- 退出群聊后该会话不再展示。

### 6.6 群聊

需求描述：

群聊功能支持多个用户在同一会话中通信，并提供基础成员管理能力。

输入：

- 群聊名称
- 初始成员
- 待添加成员
- 当前用户身份
- 群聊会话标识

输出：

- 群聊会话
- 群成员列表
- 成员操作结果
- 群消息广播结果

业务规则：

- 创建者默认成为群主。
- 群主可以添加成员。
- 普通成员可以退出群聊。
- 群主退出后应保持群聊状态可处理。
- 最后一名成员退出后，群聊可被清理。
- 非成员不能发送、查看或操作群聊资源。

异常情况：

- 非群主添加成员应被拒绝。
- 重复添加成员应稳定返回，不应产生重复关系。
- 非成员退出群聊应被拒绝。
- 用户退出后应忽略该群聊后续残留实时消息。

验收标准：

- 用户可以创建群聊。
- 群主可以添加成员。
- 群成员可以发送和接收群消息。
- 成员退出后前端状态和服务端状态一致。

### 6.7 文件消息

需求描述：

文件消息功能支持用户在会话中上传文件，并让有权限的会话成员下载。

输入：

- 当前用户身份
- 会话标识
- 文件基本信息
- 文件内容
- 文件校验信息

输出：

- 上传结果
- 文件消息
- 下载结果

业务规则：

- 上传者必须是会话成员。
- 文件上传完成后应生成可展示的文件消息。
- 下载者必须具备会话访问权限。
- 文件内容应经过基础校验，避免错误文件状态进入消息流。

异常情况：

- 非成员上传或下载应被拒绝。
- 上传过程不完整时不能生成有效文件消息。
- 文件校验失败时应拒绝完成上传。
- 文件不存在或已失效时应返回明确失败。

验收标准：

- 用户可以在会话中上传文件。
- 上传完成后聊天记录出现文件消息。
- 会话成员可以下载文件。
- 非会话成员不能下载文件。

### 6.8 前端交互

需求描述：

前端负责将后端能力组织成可演示、可操作的聊天体验。

输入：

- 用户操作
- HTTP 接口返回
- WebSocket 事件
- 本地登录状态

输出：

- 登录页
- 聊天页
- 会话列表
- 消息列表
- 成员列表
- 连接状态提示
- 错误提示

业务规则：

- 未登录用户应进入登录流程。
- 登录后进入聊天主界面。
- 页面刷新后尽量恢复登录态和当前会话。
- 当前会话切换时应加载对应历史消息。
- WebSocket 状态变化应反馈给用户。
- 前端展示应以真实接口数据为准。

异常情况：

- 登录失败时展示错误提示。
- 网络或接口失败时不应导致页面不可恢复。
- WebSocket 离线时应限制消息发送。
- 当前会话失效时应清理本地选中状态。

验收标准：

- 用户可以通过浏览器完成登录和聊天。
- 可以创建私聊、创建群聊、查看成员、发送消息和上传文件。
- 刷新后可以恢复基础状态。
- 断线重连状态可见，恢复后数据能够补偿。

## 7. 非功能需求分析

### 7.1 安全性

需求内容：

- 密码必须加密存储。
- 受保护接口必须进行身份认证。
- 会话、消息、文件等资源必须进行成员权限校验。
- 日志中不应输出明文密码等敏感信息。
- 文件下载需要校验用户是否具备访问权限。

验收重点：

- 未登录用户无法访问受保护资源。
- 非成员无法访问会话、消息和文件。
- 错误密码无法登录。

### 7.2 可靠性

需求内容：

- 消息发送采用先持久化再推送的策略。
- WebSocket 断线后前端应自动重连。
- 重连后应重新拉取会话和历史消息补偿状态。
- 空会话、离线用户、重复成员等边界情况应稳定处理。

验收重点：

- 服务重启或网络短暂异常后，用户可以恢复聊天状态。
- 离线接收方不影响发送方消息保存。
- 刷新页面后历史消息仍可查看。

### 7.3 可维护性

需求内容：

- 项目需求、接口、数据库、阶段总结应分别记录。
- 需求文档不混入接口细节和实现细节。
- phase 文档记录每个阶段的目标、变更、验证结果和后续建议。
- 错误响应和日志应便于定位问题。

验收重点：

- 新读者可以通过文档理解项目目标和范围。
- 开发者可以通过 API 文档进行联调。
- 开发者可以通过 summary 文档理解阶段演进。

### 7.4 可扩展性

需求内容：

- 会话模型应支持私聊和群聊。
- 成员角色为后续群权限扩展保留空间。
- 文件消息为后续图片、附件等消息类型扩展提供基础。
- WebSocket 和消息链路为后续在线状态、已读回执、消息状态扩展提供基础。

验收重点：

- 新增消息类型或群管理规则时，不需要推翻当前核心模型。
- 后续引入 Redis、对象存储或部署能力时，当前需求边界清晰。

### 7.5 可测试性

需求内容：

- 后端应具备单元测试、路由测试和真实数据库集成测试。
- HTTP API 应可通过 Postman 或 curl 验证。
- WebSocket 应可通过双端连接验证实时消息。
- 前端应通过浏览器真实操作验收。

验收重点：

- 核心聊天链路有自动化测试覆盖。
- 真实数据库链路可以独立验证。
- 前端演示流程可以重复操作。

## 8. 数据与接口需求概述

本节只概述数据与接口范围，不重复 `docs/api/http-api.md` 中的请求体、响应体、错误码和调试示例。

### 8.1 核心数据范围

RustChat 当前数据需求围绕以下对象展开：

- 用户：保存账号、密码哈希、头像等基础信息。
- 会话：统一表示私聊和群聊。
- 会话成员：维护用户与会话的成员关系和基础角色。
- 私聊用户对：保证同一对用户之间私聊唯一。
- 消息：保存文本、文件等会话消息。
- 已读状态：记录用户在会话中的阅读进度。
- 文件：保存文件上传后的元信息和访问所需状态。

详细表结构、约束、索引和迁移说明以 `docs/architecture/db.md` 与 migration 文件为准。

### 8.2 HTTP 接口范围

HTTP 接口主要覆盖以下能力类别：

- 健康检查与数据库连通性检查。
- 用户注册、登录、当前用户信息和用户搜索。
- 私聊和群聊会话创建。
- 群成员添加、查询与退出。
- 会话标记已读。
- 会话列表查询。
- 历史消息查询。
- 文件上传、上传完成和文件下载。

完整接口路径、参数、响应结构和错误说明以 `docs/api/http-api.md` 为准。

### 8.3 WebSocket 能力范围

WebSocket 主要覆盖以下需求：

- 登录用户建立实时连接。
- 服务端返回连接成功状态。
- 客户端发送聊天消息。
- 服务端返回发送确认。
- 服务端向在线会话成员推送消息。
- 服务端返回错误事件。
- 前端根据连接状态进行重连和数据补偿。

WebSocket 事件格式与调试方式以 `docs/api/http-api.md` 和 phase summary 中的验收说明为准。

## 9. 范围边界

### 9.1 必做范围

当前项目的基础必做范围包括：

- 用户注册和登录
- JWT 身份认证
- 私聊会话
- 群聊会话
- 文本消息发送
- WebSocket 实时通信
- 消息持久化
- 历史消息查询
- 会话列表
- 未读数基础统计
- 基础前端聊天界面
- Postman 或浏览器联调验证

### 9.2 已增强范围

在基础需求之外，当前项目已经补强的能力包括：

- 用户搜索
- 会话标记已读
- 私聊唯一性约束
- 群成员添加和退出
- 群主基础角色
- WebSocket 自动重连
- 重连后的数据补偿
- 刷新恢复当前会话
- 文件上传和下载
- 前端真实数据展示
- 真实数据库集成测试
- phase summary 阶段记录

### 9.3 暂不做范围

为控制项目复杂度，以下能力不纳入当前版本需求：

- 音视频通话
- 语音消息
- 朋友圈或动态流
- 端到端加密
- 多端同步
- 多窗口连接协调
- 消息撤回或编辑
- 已读回执的完整实时同步
- 离线消息发送队列和自动补发
- 复杂群权限、管理员、多群主和群主转让完整流程
- 分布式部署
- Redis 在线状态
- MQ 消息队列
- 对象存储接入
- 商业级风控、审计和大规模高并发优化

这些内容可以作为后续演进方向，但不作为当前需求验收条件。

## 10. 测试与验收标准

### 10.1 Postman HTTP 验收

验收目标：

- 验证注册、登录、用户搜索、会话创建、会话列表、历史消息、群成员、文件上传下载等 HTTP 能力。

验收标准：

- 受保护接口必须携带合法登录凭证。
- 正常业务操作返回成功结果。
- 权限错误、参数错误和资源不存在等场景能够返回明确失败。
- 会话列表、历史消息和已读状态能够反映实际业务变化。

### 10.2 WebSocket 双端验收

验收目标：

- 验证私聊和群聊实时消息链路。

验收标准：

- 两个用户分别建立 WebSocket 连接后，可以完成私聊实时收发。
- 多个群成员建立连接后，可以收到群消息广播。
- 发送方收到发送确认。
- 接收方在线时收到实时消息。
- 非法 token 不能建立有效连接。

### 10.3 前端实机验收

验收目标：

- 验证用户能通过浏览器完成完整聊天操作。

验收标准：

- 用户可以注册或登录。
- 用户可以搜索其他用户并创建私聊。
- 用户可以创建群聊、添加成员和退出群聊。
- 用户可以发送文本消息和文件消息。
- 用户可以查看会话列表、成员列表和历史消息。
- 用户刷新页面后可以恢复基础状态。
- WebSocket 断开和重连状态对用户可见。

### 10.4 真实数据库测试

验收目标：

- 验证核心链路在真实 PostgreSQL 环境中可重复运行。

验收标准：

- migration 可以正确执行。
- 注册、登录、私聊、群聊、消息、会话列表和已读状态能够在真实库中完成。
- 测试库与开发库隔离。
- 测试可以重复执行，不依赖人工清理日常开发数据。

### 10.5 操作视频或演示验收

验收目标：

- 用可视化方式证明项目具备可演示闭环。

建议演示流程：

- 用户 A 和用户 B 登录。
- 用户 A 搜索用户 B 并创建私聊。
- 双方实时发送消息。
- 刷新页面后查看历史消息。
- 用户 A 创建群聊并添加用户 B。
- 群聊中发送消息并查看成员列表。
- 上传文件并下载。
- 模拟 WebSocket 断线后恢复连接。

验收标准：

- 演示过程不依赖手动修改数据库。
- 核心功能可以稳定复现。
- 异常状态有明确提示，不出现明显流程中断。

## 11. 总结

RustChat 当前已经从一个 Rust Web 后端练手项目，演进为具备前后端、数据库、WebSocket、文件消息和联调验收资料的轻量级即时聊天系统。

从需求角度看，项目已经完成以下闭环：

- 用户身份闭环：注册、登录、鉴权、当前用户识别。
- 聊天会话闭环：私聊、群聊、成员关系、会话列表。
- 消息通信闭环：消息发送、持久化、实时推送、历史查询。
- 前端操作闭环：登录、进入聊天页、切换会话、发送消息、查看历史。
- 验收验证闭环：API 文档、Postman、WebSocket 验证、真实数据库测试、phase 总结。

因此，RustChat 当前版本可以被认为已经形成“可运行、可演示、可验证”的基础聊天系统。后续如果继续扩展，应优先在当前清晰边界之上选择单点增强，避免偏离轻量级 IM 和软件工程实践项目的核心定位。
