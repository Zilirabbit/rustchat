# RustChat 需求分析与可行性分析文档

## 一、需求分析

### 1. 项目背景

RustChat 是一个基于 Rust 后端和 Web 前端实现的轻量级即时聊天系统，项目定位为课程作业，系统目标不是构建商业级 IM 产品，而是围绕 **基础聊天闭环** 完成一个可运行、可演示、可验证的 Web 即时通信应用。

项目当前以 Rust + axum + tokio + sqlx + PostgreSQL 作为后端技术基础，以 Vue 3 + Pinia + Vue Router + Vite 作为前端技术基础，采用 HTTP API 与 WebSocket 混合通信方式。HTTP API 负责用户、会话、历史消息、文件上传等请求型业务，WebSocket 负责实时消息收发和在线推送。

从项目演进看，RustChat 已经经历了基础工程、用户鉴权、私聊链路、会话列表、历史消息、前后端联调、群聊、WebSocket 重连、真实数据展示、文件上传稳定化等阶段。相关阶段过程记录在 `docs/summary/` 中，接口细节记录在 `docs/api/http-api.md` 中，数据库设计记录在 `docs/architecture/db.md` 中。

项目建设目标包括：

- 完成用户注册、登录和身份认证。
- 完成私聊、群聊、消息发送和历史消息查询。
- 完成 WebSocket 实时通信与断线重连体验。
- 完成会话列表、未读状态和成员管理等基础 IM 能力。
- 完成文件、图片、GIF/表情、语音等媒体消息的基础支持。
- 提供可通过浏览器操作的前端聊天界面。
- 通过文档、任务拆解、phase 总结和验收记录体现软件工程过程。

### 2. 用户角色分析

当前版本只面向一种核心业务角色：普通用户。

普通用户的主要需求包括：

- 注册账号并登录系统。
- 搜索其他用户并创建私聊。
- 创建群聊、查看群成员、添加或移除成员。
- 在私聊或群聊中发送文本、文件、图片、GIF/表情和语音消息。
- 实时接收在线消息。
- 查看会话列表、最近消息和未读状态。
- 查看历史消息。
- 在网络异常或页面刷新后恢复聊天状态。

普通用户在不同场景下的权限有所区分：

- 未登录用户只能访问注册、登录和基础健康检查类能力。
- 已登录用户可以访问自己的会话、消息、文件和群成员信息。
- 私聊成员只能访问自己参与的私聊。
- 群聊成员可以查看群聊消息和成员列表。
- 群聊 owner 可以添加或移除普通成员。
- 非会话成员不能发送消息、查看历史消息或下载会话文件。

除普通用户外，项目还存在开发、测试和验收人员，但这些不是业务功能角色，主要用于本地运行、接口验证、WebSocket 验收、真实数据库测试和演示录制。

### 3. 功能需求

#### 3.1 用户与认证

需求描述：

- 支持用户注册、登录和获取当前登录用户信息。
- 支持基于 JWT 的身份认证。
- 支持用户搜索，为创建私聊和群聊添加成员提供前置能力。

业务规则：

- 用户名必须唯一。
- 密码必须加密存储，不能明文保存。
- 受保护接口必须校验登录态。
- 用户搜索只返回聊天创建所需的基础公开信息。

验收标准：

- 新用户可以注册。
- 已注册用户可以登录并获得有效登录凭证。
- 登录后可以访问受保护功能。
- 未登录或 token 无效时无法访问受保护资源。
- 用户可以通过关键词搜索其他用户。

#### 3.2 会话系统

需求描述：

- 支持私聊和群聊两类会话。
- 支持维护用户与会话之间的成员关系。
- 支持会话已读状态更新。

业务规则：

- 用户不能与自己创建私聊。
- 同一对用户之间应保持唯一私聊会话。
- 群聊必须有创建者，创建者默认为 owner。
- 会话访问必须基于成员关系校验。
- 用户只能操作自己有权限的会话。

验收标准：

- 用户可以创建或复用私聊会话。
- 用户可以创建群聊会话。
- 系统可以正确维护会话成员关系。
- 非成员无法访问会话消息或操作会话资源。
- 用户打开会话后可以更新已读状态。

#### 3.3 消息系统

需求描述：

- 支持在私聊和群聊中发送消息。
- 支持文本消息、文件消息、图片/GIF 消息和语音消息。
- 支持消息持久化和历史消息分页查询。

业务规则：

- 只有会话成员可以发送消息。
- 文本消息内容不能为空。
- 消息应先保存到数据库，再进行实时推送。
- 历史消息查询必须进行成员权限校验。
- 文件类消息应在上传完成后生成消息记录。

验收标准：

- 私聊和群聊中均可发送文本消息。
- 文件、图片/GIF、语音等媒体消息可进入聊天记录。
- 在线成员可以实时收到消息。
- 页面刷新后仍可查询历史消息。
- 非成员无法发送或查询消息。

#### 3.4 实时通信

需求描述：

- 支持登录用户建立 WebSocket 连接。
- 支持通过 WebSocket 发送消息、接收消息和获得发送确认。
- 支持连接状态展示和自动重连。

业务规则：

- WebSocket 建立连接时必须完成身份校验。
- 无效 token 不能建立有效连接。
- 消息发送仍需复用会话成员权限校验。
- 前端只应在连接可用时允许发送实时消息。
- 重连成功后应重新拉取会话列表和当前会话历史消息，以补偿断线期间可能遗漏的数据。

验收标准：

- 合法用户可以建立 WebSocket 连接。
- 私聊消息可以实时推送给在线接收方。
- 群聊消息可以广播给在线群成员。
- 连接断开后前端能够显示状态变化并尝试重连。
- 重连后会话和消息状态能够恢复。

#### 3.5 会话列表与未读状态

需求描述：

- 支持展示当前用户参与的会话列表。
- 支持展示最近消息、最近活跃时间和未读数量。
- 支持进入会话后更新已读状态。

业务规则：

- 会话列表只展示当前用户所属会话。
- 会话应按最近活跃时间排序。
- 未读数按“用户 + 会话”维度统计。
- 用户退出群聊后，该群聊应从该用户会话列表中移除。

验收标准：

- 用户登录后可以看到自己的会话列表。
- 发送新消息后最近消息和排序会更新。
- 用户打开会话后未读数可归零。
- 退出群聊后前端状态与服务端状态保持一致。

#### 3.6 群聊与成员管理

需求描述：

- 支持创建群聊。
- 支持群 owner 添加成员、移除普通成员。
- 支持普通成员主动退出群聊。
- 支持查看群成员列表。

业务规则：

- 群聊创建者自动成为 owner。
- 只有 owner 可以添加或移除成员。
- owner 不能通过移除成员接口移除自己。
- 普通成员可以主动退出群聊。
- 非成员不能查看群成员列表、发送群消息或查询群历史。

验收标准：

- 用户可以创建群聊并添加初始成员。
- owner 可以添加新成员。
- owner 可以移除普通成员。
- 普通成员可以主动退出群聊。
- 成员变更后会话列表、成员列表和实时消息状态保持一致。

#### 3.7 文件与媒体消息

需求描述：

- 支持文件分片上传、上传完成确认和下载。
- 支持图片/GIF 在消息列表中预览。
- 支持语音消息在消息列表中播放。
- 支持保存图片/GIF 作为本地自定义表情。

业务规则：

- 上传用户必须是目标会话成员。
- 文件上传过程需要校验上传发起人、文件元数据、分片完整性和文件哈希。
- 上传完成后，文件记录、消息记录和会话最近消息状态应保持一致。
- 下载文件时必须校验用户是否仍具备会话访问权限。

验收标准：

- 用户可以在会话中上传文件。
- 文件上传完成后会生成文件消息。
- 图片/GIF 可以预览，语音可以播放。
- 会话成员可以下载文件。
- 非会话成员不能下载文件。

#### 3.8 前端交互

需求描述：

- 提供登录页和聊天页。
- 支持会话列表、消息列表、消息输入、群成员列表和连接状态展示。
- 支持刷新后恢复登录态和当前会话。
- 支持上传进度、录音状态、连接状态和错误提示。

业务规则：

- 未登录用户应进入登录流程。
- 登录后进入聊天主界面。
- 前端展示应以真实接口数据为准。
- 当前会话失效或用户退出群聊后，应清理本地会话状态。
- WebSocket 离线时应限制实时消息发送。

验收标准：

- 用户可以通过浏览器完成登录、创建会话、发送消息和查看历史。
- 用户可以上传文件、发送图片/GIF 和录制语音。
- 用户可以查看群成员并进行权限允许范围内的成员操作。
- 页面刷新后可以恢复基础聊天状态。
- 网络异常时有明确连接状态提示。

### 4. 非功能需求

#### 4.1 安全性

- 密码必须加密存储。
- 受保护接口必须进行身份认证。
- 会话、消息、文件和群成员操作必须进行权限校验。
- 日志中不应记录明文密码等敏感信息。
- 文件下载必须校验访问者是否属于对应会话。

#### 4.2 可靠性

- 消息发送应采用先落库再推送的策略。
- 离线接收方不应阻塞发送方消息保存。
- WebSocket 断开后前端应尝试自动重连。
- 重连成功后应重新拉取关键数据，降低漏消息影响。
- 空会话、重复成员、退出群聊等边界场景应稳定处理。

#### 4.3 可维护性

- 需求、接口、数据库、架构、阶段总结应分文档维护。
- 接口细节集中维护在 `docs/api/http-api.md`。
- 数据设计集中维护在 `docs/architecture/db.md`。
- 阶段演进与验证记录集中维护在 `docs/summary/`。
- 业务模块应保持职责清晰，便于后续扩展和排错。

#### 4.4 可扩展性

- 会话模型应同时支持私聊和群聊。
- 成员角色应为后续更复杂的群权限保留空间。
- 消息类型应支持从文本扩展到文件、图片、GIF、语音等媒体类型。
- 文件存储当前可使用本地目录，后续可扩展到对象存储。
- 在线状态当前可基于进程内连接管理，后续可扩展 Redis 等外部状态服务。

#### 4.5 性能与可用性

- 学习项目阶段应支持本地和局域网环境下的基础多人聊天演示。
- 常规 HTTP 接口响应应满足浏览器交互需要。
- 历史消息应采用分页方式，避免一次加载过多数据。
- 文件上传应支持分片，避免大文件一次性提交造成体验和资源问题。
- 前端应提供清晰的 loading、上传进度、连接状态和错误反馈。

#### 4.6 可测试性

- 后端应具备单元测试、路由测试和真实数据库集成测试。
- HTTP 接口应能通过 Postman 或 curl 验证。
- WebSocket 应能通过双端或多端连接验证实时消息。
- 前端应能通过浏览器实机操作验收。
- 项目应保留 phase summary，记录每阶段验证结果和遗留问题。

### 5. 数据需求

RustChat 的数据需求围绕用户、会话、成员、消息、已读状态和文件展开。

核心数据对象包括：

- 用户数据：保存用户名、密码哈希、头像等账号基础信息。
- 会话数据：统一保存私聊和群聊会话信息。
- 会话成员数据：保存用户与会话的成员关系和角色。
- 私聊唯一关系数据：保证同一对用户之间私聊唯一。
- 消息数据：保存文本、文件、图片/GIF、语音等消息内容或元信息。
- 已读状态数据：记录用户在某个会话中的阅读进度。
- 文件数据：保存上传文件的名称、大小、类型、哈希、存储路径和过期清理状态。

数据规则包括：

- 用户名应保持唯一。
- 私聊用户对应保持唯一。
- 会话成员不能重复。
- 消息必须归属于某个会话。
- 文件消息必须关联上传者和会话。
- 历史消息查询应支持分页。
- 已读状态应按用户和会话维度维护。

数据持久化要求：

- 用户、会话、成员、消息、已读状态和文件元信息应保存到 PostgreSQL。
- 数据库结构变更应通过 migration 管理。
- 文件内容当前可存储在本地上传目录，文件元信息保存在数据库中。
- 真实数据库测试应使用独立测试库，避免污染开发库。

### 6. 接口需求

RustChat 的接口需求分为 HTTP API 和 WebSocket 两类。

HTTP API 主要承担请求型业务：

- 服务健康检查和数据库连通检查。
- 用户注册、登录、当前用户信息和用户搜索。
- 私聊创建、群聊创建、会话成员管理和已读状态更新。
- 会话列表查询和历史消息查询。
- 文件上传初始化、分片上传、上传完成和文件下载。

WebSocket 主要承担实时通信业务：

- 登录用户建立实时连接。
- 客户端发送消息。
- 服务端返回发送确认。
- 服务端向在线成员推送私聊或群聊消息。
- 服务端返回连接或业务错误事件。
- 前端根据连接状态进行自动重连和数据补偿。

接口设计要求：

- 受保护 HTTP 接口需要身份认证。
- WebSocket 连接需要身份认证。
- 会话、消息、成员、文件接口需要校验资源权限。
- 接口响应格式应保持统一。
- 前端应基于接口返回状态展示成功、失败、loading 和错误提示。

完整接口路径、参数、响应体、错误码和调试示例不在本文档重复展开，以 `docs/api/http-api.md` 为准。

### 7. 验收标准

#### 7.1 基础功能验收

- 用户可以注册、登录并访问受保护功能。
- 用户可以搜索其他用户。
- 用户可以创建私聊并发送消息。
- 用户可以创建群聊、添加成员、移除成员和退出群聊。
- 用户可以查看会话列表、最近消息、未读数和历史消息。
- 用户可以上传和下载文件。
- 用户可以发送图片/GIF 和语音消息。

#### 7.2 实时通信验收

- 合法用户可以建立 WebSocket 连接。
- 私聊双方在线时可以实时收发消息。
- 群聊成员在线时可以实时收到群消息。
- 发送方可以收到发送确认。
- 非法 token 或非成员发送行为会被拒绝。
- WebSocket 断开后前端能展示状态并尝试重连。

#### 7.3 数据一致性验收

- 消息发送后可以在历史消息中查询到。
- 会话列表最近消息和排序能随新消息更新。
- 用户进入会话并标记已读后，未读数能够更新。
- 群成员变更后，成员列表和会话访问权限同步生效。
- 文件上传完成后，文件记录、消息记录和会话最近消息状态一致。

#### 7.4 前端演示验收

- 可以通过浏览器完成登录和聊天主流程。
- 可以通过两个浏览器窗口演示私聊实时消息。
- 可以通过多个用户演示群聊消息广播。
- 可以演示文件上传、图片/GIF 预览、语音播放。
- 可以演示刷新恢复当前会话。
- 可以演示断线重连后的数据恢复。

#### 7.5 测试与文档验收

- Postman collection 可用于 HTTP 和 WebSocket 基础验证。
- 后端常规测试可运行通过。
- 真实数据库 ignored 测试可在独立测试库中验证。
- 项目文档能够说明产品范围、接口、数据库、环境搭建和阶段演进。
- 操作视频或课堂演示能够覆盖核心聊天闭环。

## 二、可行性分析

### 1. 技术可行性

RustChat 的技术方案具备较高可行性。

后端使用 Rust、axum、tokio、sqlx 和 PostgreSQL，能够覆盖 Web 服务、异步处理、数据库访问、migration 管理和 WebSocket 通信等核心需求。Rust 在类型安全和内存安全方面优势明显，适合实践高可靠服务端开发；axum 与 tokio 生态可以满足 HTTP + WebSocket 混合服务模型；sqlx 与 PostgreSQL 可以支持用户、会话、消息和文件元信息的持久化。

前端使用 Vue 3、Pinia、Vue Router、Vite 和 axios，能够满足单页聊天应用的界面、路由、状态管理、接口调用和构建需求。当前前端已经具备登录页、聊天页、会话列表、消息列表、消息输入、连接状态、文件上传和语音录制等基础交互，说明技术栈能够支撑项目目标。

实时通信方面，WebSocket 适合聊天系统的在线消息推送需求。当前系统采用 HTTP 处理请求型业务、WebSocket 处理实时消息的方式，职责边界清晰，符合轻量级 IM 项目的技术特点。

文件与媒体消息方面，当前方案采用本地 `UPLOAD_DIR` 保存文件内容，数据库保存文件元信息，适合课程和练手项目阶段。后续如果扩展到生产形态，可以替换为对象存储或 CDN，但这不影响当前版本的技术可行性。

技术风险主要包括：

- Rust 学习成本高于常见脚本语言。
- WebSocket 调试和异常状态处理比普通 HTTP 更复杂。
- 文件上传涉及分片、哈希、权限和清理，需要更多边界测试。
- 前端录音和 Web Crypto 等浏览器能力存在兼容性差异。

总体判断：现有技术选型已经在项目中落地，核心链路可运行，技术可行。

### 2. 经济可行性

RustChat 使用的主要技术栈均为开源技术，无需购买商业授权。开发和运行所需资源主要包括本地开发机、PostgreSQL 数据库、浏览器、Postman 或同类接口调试工具。

当前版本采用本地文件存储，不依赖对象存储、短信服务、推送服务、商业 IM SDK 或云厂商托管服务，因此成本较低。对于课程作业、个人练手和软件工程实践而言，经济投入主要是开发时间和本地环境维护成本。

经济成本主要包括：

- 开发者学习 Rust、WebSocket 和前后端联调的时间成本。
- 本地或虚拟机运行 PostgreSQL 的环境成本。
- 文件上传带来的本地磁盘占用。
- 若进行演示视频录制，需要少量录屏和测试数据准备时间。

与项目目标相比，这些成本可控，且能换取完整的软件工程实践经验。

总体判断：项目不依赖高额商业服务，经济可行。

### 3. 操作可行性

从用户操作角度看，RustChat 采用浏览器访问方式，不需要用户安装复杂客户端。用户只需完成注册、登录，即可进入聊天界面，进行私聊、群聊、消息发送和文件上传等操作。

从开发和测试角度看，项目已经提供 API 文档、Postman collection、环境搭建文档、phase summary 和问题记录，便于开发者复现接口调用、定位联调问题和完成验收。前端基于真实接口展示，能够减少“页面 mock 与实际能力不一致”的问题。

操作可行性优势包括：

- 浏览器即可访问前端页面。
- Postman 可验证 HTTP 和 WebSocket 基础链路。
- 文档按产品、接口、架构、环境、summary 分类，查找路径清晰。
- phase summary 记录了阶段成果和已知问题，便于后续维护。
- 项目核心演示流程可以通过两个或多个浏览器窗口完成。

操作风险主要包括：

- 首次运行需要正确配置数据库和环境变量。
- WebSocket 验收需要准备多个用户和多个浏览器窗口。
- 文件上传、语音录制等能力受浏览器权限和本地环境影响。
- 如果开发库和测试库不隔离，真实数据库测试可能污染数据。

总体判断：系统操作路径清晰，配套文档较完整，操作可行。

### 4. 时间可行性

从阶段记录看，RustChat 已经完成基础工程、用户认证、私聊、会话列表、历史消息、真实数据库测试、前端联调、群聊、WebSocket 重连、真实数据展示和文件上传稳定化等阶段。当前项目不是从零开始，而是在已有实现基础上进行文档整理、验收补强和局部完善。

对于课程作业或软件工程实践项目，当前范围是可控的：

- 核心聊天闭环已经形成。
- 主要接口和数据库结构已经存在。
- 前端基础页面和交互已经完成。
- 关键阶段已有 summary 记录。
- 剩余工作主要集中在补充验收、修正文档、整理演示和处理已知问题。

时间风险主要包括：

- 如果继续扩大范围，例如加入分布式部署、复杂群权限、完整多端同步，会显著增加时间成本。
- 媒体消息和浏览器录音能力存在兼容性排查成本。
- WebSocket 异常路径和重连补偿需要更多手动验收时间。
- 真实数据库集成测试需要稳定的独立测试库。

控制方式：

- 当前阶段应以“完成演示闭环和验收材料”为主。
- 不再扩展音视频、复杂权限、分布式、高并发等非目标能力。
- 将遗留问题记录到 `docs/problems/` 或 phase summary 中。

总体判断：在控制范围的前提下，时间可行。

### 5. 安全与风险分析

#### 5.1 安全可控点

当前项目已经具备基础安全措施：

- 密码使用哈希方式存储。
- HTTP 受保护接口通过 JWT 鉴权。
- WebSocket 连接需要 token。
- 会话、消息、成员和文件访问基于成员关系校验。
- 文件上传完成时进行基础元数据、大小和哈希校验。
- 非成员不能访问对应会话消息和文件。

这些措施能够满足课程和练手项目阶段的基础安全要求。

#### 5.2 主要风险

项目仍存在一些需要关注的风险：

- token 存储在前端时需要注意 XSS 风险。
- 本地文件存储需要防止路径穿越、文件残留和磁盘占用失控。
- 文件 MIME 类型和扩展名不能完全代表真实文件内容。
- WebSocket 长连接需要处理异常断开、重复连接和资源清理。
- CORS 配置在联调环境和未来部署环境中需要谨慎设置。
- 真实数据库测试如果配置错误，可能误操作开发库数据。
- 当前系统不具备商业级风控、审计、限流和多租户隔离能力。

#### 5.3 风险应对

建议采取以下控制措施：

- 保持输入校验和权限校验覆盖所有受保护资源。
- 文件下载只允许会话成员访问。
- 上传目录与应用代码目录隔离。
- 对上传文件大小、分片数量和过期时间进行限制。
- 将开发库和测试库明确分离。
- 对已知问题建立问题文档并保留复现条件。
- 演示和验收环境只开放必要服务。

总体判断：当前风险在学习项目范围内可控，但不应将其直接视为生产级安全方案。

### 6. 可行性结论

综合技术、经济、操作、时间和安全风险分析，RustChat 项目具备较高可行性。

技术上，Rust + axum + tokio + sqlx + PostgreSQL 能够支撑后端核心能力，Vue 3 + Pinia + Vite 能够支撑前端交互，HTTP + WebSocket 的组合适合轻量级聊天系统。经济上，项目主要依赖开源技术和本地环境，不需要额外商业服务。操作上，浏览器、Postman、文档和 phase summary 可以支撑开发、测试和演示。时间上，项目核心链路已经完成，剩余工作主要是验收、文档和局部稳定性补强。安全上，项目具备基础鉴权、密码哈希、成员权限和文件访问控制，风险在课程和练手项目范围内可控。

因此，RustChat 作为课程作业、个人练手和软件工程实践项目是可行的。当前版本已经具备“可运行、可演示、可验证”的基础聊天系统特征。后续应继续围绕稳定性、验收材料和文档一致性进行收口，避免在当前阶段引入音视频通话、复杂权限、多端同步、分布式部署和商业级高并发等超出项目定位的能力。
