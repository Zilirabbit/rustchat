# RustChat - 产品规格文档（refined spec.md）

## 1. 文档信息

- 项目名称：RustChat
- 项目类型：课程作业 / 练手项目 / 软件工程流程实践项目
- 当前版本：v1.0
- 文档目的：明确产品范围、核心功能、模块边界与验收标准，作为开发与测试的统一依据

---

## 2. 项目概述

RustChat 是一个基于 **Rust + WebSocket** 的 Web 即时通信系统，支持用户注册登录、私聊、群聊、会话列表、历史消息查询，并具备基础的消息持久化能力。

本项目的目标不是做一个商业级 IM，而是：

1. 完整走通一套软件工程开发流程
2. 熟悉 Rust Web 后端开发
3. 掌握 WebSocket 实时通信模型
4. 建立清晰、可维护、可扩展的后端模块化结构

---

- 能通过文档、任务拆解、阶段总结来管理开发过程

项目最终至少应交付：

- 基础接口文档

## 4. 用户角色与使用场景

### 4.1 用户角色

| 角色 | 描述 |
|------|------|
| 普通用户 | 注册、登录、创建会话、发送消息、查看会话与历史消息 |

### 4.2 核心使用场景

#### 场景 1：用户注册与登录

用户通过用户名和密码完成注册，之后可登录获取 JWT token。

#### 场景 2：用户建立私聊并发送消息

用户 A 与用户 B 建立私聊会话，通过 WebSocket 发送文本消息；若对方在线，则实时接收；若对方离线，则消息保存在数据库中。

#### 场景 3：用户进入群聊并接收群消息

用户加入某个群聊后，可向群聊发送消息，群成员在线时可实时收到。

#### 场景 4：用户查看会话列表与历史消息

用户登录后可查看自己的会话列表，展示最近消息与未读数；进入会话后可分页加载历史消息。

## 6. 功能需求

## 6.1 用户系统

### 功能

- 用户注册
- 用户登录
- 获取当前用户信息

### 输入

- username
- password

### 输出

- user_id
- username
- JWT token（登录成功时）

### 业务规则

- 用户名唯一
- 用户名长度需限制（建议 3~20）
- 密码长度需限制（建议 6~32）
- 密码必须加密存储，不可明文保存
- 登录失败时返回统一错误信息

### 验收标准

- 可成功注册新用户
- 已存在用户名不可重复注册
- 正确账号密码可登录成功
- 错误密码不可登录
- 登录成功后返回 token

---

## 6.2 会话系统（Session）

### 类型

- 私聊（private）
- 群聊（group）

### 核心能力

- 创建私聊会话
- 创建群聊会话
- 查询用户所属会话
- 维护会话成员关系

### 数据结构建议

```text
Session {
  id
  type            // private | group
  name            // 群聊可有名称，私聊可为空
  avatar          // 可选
  created_by
  created_at
}
```

### 业务规则

- 私聊会话成员固定为 2 人
- 同一对用户之间可约束只存在一个私聊会话（建议）
- 群聊可包含多个成员
- 非会话成员不可发送该会话消息

### 验收标准

- 可成功创建私聊
- 可成功创建群聊
- 可正确查询会话成员
- 非成员发送消息会被拒绝

---

## 6.3 会话列表（Conversation）

### 功能

- 获取用户会话列表
- 展示最近一条消息
- 展示未读消息数
- 按最近活跃时间排序

### 展示信息建议

```text
ConversationItem {
  session_id
  session_type
  session_name
  last_message
  last_message_time
  unread_count
}
```

### 业务规则

- 会话按最近消息时间倒序排列
- 若会话暂无消息，也可展示但排在后面
- 未读数按“当前用户 + 会话”维度统计

### 验收标准

- 用户可获取自己的会话列表
- 列表中可展示最近消息与时间
- 未读数可正确显示

---

## 6.4 消息系统（Message）

### 消息类型

- 文本（v1 必做）
- 图片（可选）

### 数据结构建议

```text
Message {
  id
  session_id
  sender_id
  content
  type
  created_at
}
```

### 核心流程

```text
Client -> WebSocket -> message handler
       -> 权限校验
       -> DB 持久化
       -> 推送给会话在线成员
```

### 业务规则

- 只有登录用户才能发送消息
- 只有会话成员才能发送消息
- 消息发送成功后必须先落库，再推送
- 文本消息内容不能为空
- 可限制最大消息长度（建议 1000 字符）

### 验收标准

- 用户可在私聊中发送文本消息
- 用户可在群聊中发送文本消息
- 消息会保存到数据库
- 在线成员可实时收到消息
- 刷新页面后仍可查询历史消息

---

## 6.5 实时通信（WebSocket）

### 能力

- 建立 WebSocket 连接
- 连接时携带 JWT 并完成鉴权
- 维护 user_id 与连接对象的映射
- 单用户推送
- 会话广播
- 断线清理
- 心跳检测（可选）

### 业务规则

- 无效 token 不允许建立连接
- 连接断开后需清理在线状态映射
- 可允许同一用户多端连接，但当前版本可先只支持单连接模型

### 验收标准

- 合法用户可建立 WS 连接
- 非法 token 无法连接
- 连接断开后服务端状态正确清理
- 在线用户可收到实时消息

---

## 6.6 历史消息

### 功能

- 按会话查询历史消息
- 支持分页加载
- 支持按时间倒序或游标分页

### 业务规则

- 非会话成员不可查询消息历史
- 历史消息查询必须带分页参数
- 单次查询数量需限制（如 20 / 50）

### 验收标准

- 用户可查询所属会话历史消息
- 支持分页获取
- 分页结果顺序稳定

---

## 7. 非功能需求

### 7.1 性能

- 学习项目级别支持 ≥ 100 并发连接
- 常规接口响应时间应保持在可接受范围内
- WebSocket 消息链路在本地环境下表现稳定

### 7.2 安全性

- 密码加密存储
- 接口鉴权
- 基础输入校验
- 避免明文敏感信息写入日志

### 7.3 可维护性

- 模块边界清晰
- 分层明确
- 统一错误处理
- 统一日志输出

### 7.4 可扩展性

- 后续可扩展 Redis 在线状态
- 后续可扩展对象存储 / 文件服务
- 后续可扩展 MQ、Nginx、分布式部署
