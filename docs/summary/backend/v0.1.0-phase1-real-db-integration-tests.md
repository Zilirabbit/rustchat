## 关键实现说明

### 1. 为什么使用独立 database

本次最终在同一个 PostgreSQL 实例中创建独立 database：

```text
rustchat_test
```

最开始当前数据库用户 `db_rabbit` 没有 `CREATE DATABASE` 权限，所以曾临时使用过 `rustchat` database 内的 `rustchat_test` schema。随后补齐权限并成功创建独立 database 后，已切换为当前方案。

独立 database 比 schema 隔离更适合作为长期测试方案：

- 开发库 `rustchat` 与测试库 `rustchat_test` 完全分离
- migration 元数据、表、序列和测试数据都只存在于测试库
- 测试清理和重建更放心，不会影响 Postman 手工测试数据
- 后续接入 CI 时可以直接复用同样的 database 级隔离方式

### 2. 测试连接直接指向 `rustchat_test`

测试连接格式为：

```text
postgres://db_rabbit:***@192.168.221.131:5432/rustchat_test
```

这样 sqlx migration 和业务 SQL 都会默认落到独立测试 database 的 `public` schema 中。

本次已验证：

```text
current_database = rustchat_test
current_schema = public
```

真实库测试执行后，`rustchat_test` database 中会创建项目核心表和 sqlx migration 元数据。

### 3. 为什么真实库测试使用 `#[ignore]`

新增测试被标记为：

```rust
#[ignore = "requires TEST_DATABASE_URL to point at the rustchat_test database"]
```

原因是这类测试依赖外部资源：

- PostgreSQL 必须可连接
- `TEST_DATABASE_URL` 必须正确配置
- 测试 database 必须允许创建表和清理数据
- WebSocket 测试需要临时启动本地 server

因此普通开发时执行：

```bash
cargo test
```

不会强制连接数据库，只跑快速单元测试和路由测试。

需要验证真实数据库链路时再执行：

```bash
cargo test -- --ignored --test-threads=1
```

### 4. 为什么要求 `--test-threads=1`

当前 ignored 测试中有两个真实库测试：

- storage migration/ping 测试
- 私聊端到端真实库测试

私聊端到端测试启动时会在测试 database 内执行：

```sql
TRUNCATE TABLE
    user_session_read_state,
    messages,
    session_members,
    sessions,
    users
RESTART IDENTITY CASCADE
```

这个清理只发生在 `rustchat_test` database 中，但它仍然会影响同一个测试 database 内的并发测试。

因此真实库测试建议串行运行：

```bash
cargo test -- --ignored --test-threads=1
```

这样可以避免两个真实库测试同时 migration、清表或读写数据。

### 5. 端到端测试覆盖的真实路径

新增测试不是只测 service 层，而是尽量走真实应用路径：

1. 使用真实 `AppState::build`
2. 连接真实 PostgreSQL
3. 自动执行 migration
4. 创建真实 axum router
5. 通过 HTTP route 注册 Alice / Bob
6. 通过 HTTP route 登录并获取 JWT
7. 通过受保护 HTTP route 访问 `/api/me`
8. 通过受保护 HTTP route 创建私聊
9. 启动真实 axum WebSocket server
10. Alice / Bob 通过 WebSocket 建立连接
11. Alice 发送 `send_message`
12. 消息写入真实数据库
13. Alice 收到 `message_sent`
14. Bob 收到 `receive_message`
15. Bob 通过 HTTP 查询会话列表
16. Bob 通过 HTTP 查询历史消息

这覆盖了当前私聊主链路中最容易出问题的组合点：

- JWT 生成与解析
- middleware 当前用户提取
- repository SQL 与真实表结构是否匹配
- session_members 权限关系
- messages 入库
- sessions.last_message_id / last_message_at 更新
- conversation 最近消息与未读数统计
- history messages 分页查询
- WebSocket 在线推送

### 6. 测试数据清理策略

本次没有使用开发库 `rustchat` 中的真实数据，也没有依赖 Postman 手工创建的数据。

端到端测试启动后会先清空测试 database 中的核心表：

```sql
TRUNCATE TABLE
    user_session_read_state,
    messages,
    session_members,
    sessions,
    users
RESTART IDENTITY CASCADE
```

随后测试会创建带时间后缀的用户名：

```text
alice_it_<suffix>
bob_it_<suffix>
```

这样做有两个目的：

- 每次测试数据是自包含的，不依赖历史状态
- 如果某次测试中断，残留数据也容易识别
