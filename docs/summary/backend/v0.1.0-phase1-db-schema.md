## 关键实现说明

### 1. 表结构围绕 IM 主链路设计

本次表设计直接对应当前版本目标：

- `users`：支撑注册、登录与基础资料
- `sessions`：统一承载私聊与群聊
- `session_members`：维护会话成员关系
- `messages`：存储消息正文与发送者信息
- `user_session_read_state`：为未读统计与已读进度预留基础状态

### 2. 为“会话列表”和“历史消息”提前预留字段

`sessions` 中增加：

- `last_message_id`
- `last_message_at`

这样后续查询会话列表时，不需要完全依赖实时聚合消息表，能更平滑地支持最近消息展示与会话排序。

`messages` 中增加：

- `(session_id, id DESC)` 索引

用于支持按会话倒序分页查询历史消息。

### 3. 关键关系用复合外键保证“消息属于当前会话”

为了避免：

- `sessions.last_message_id` 指向别的会话消息
- `user_session_read_state.last_read_message_id` 指向别的会话消息

本次增加了基于 `(session_id, id)` 的复合约束，确保这些“指向消息”的字段和所属会话一致。

### 4. 手动导入与 migration 体系已完成对齐

由于本次先走了“手动导入数据库”的流程，如果只创建表而不登记 `_sqlx_migrations`，后续应用启动时会再次尝试执行 3.4 migration，并因表已存在而失败。

当前已补齐 `20260423010000_create_chat_core_tables` 的 migration 记录，后续可以继续沿用 `sqlx` 正常启动和迭代。

当前数据库中的 migration 记录为：

- `20260423000000 | init storage | t`
- `20260423010000 | create chat core tables | t`

### 5. 3.4 当前只保留一个正式结构来源

为了避免后续出现“双份 SQL 内容漂移”，本次已经删除临时手动导入文件，当前只保留：

- `backend/migrations/20260423010000_create_chat_core_tables.sql`

也就是说，后续无论是本地启动、联调环境初始化还是团队协作，都应以这份 migration 为准。
