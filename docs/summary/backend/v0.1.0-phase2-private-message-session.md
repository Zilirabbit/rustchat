## 关键实现说明

### 1. 私聊创建接口先收口到“最小可复用”

本次新增接口：

- `POST /api/sessions/private`

请求只需要：

- `target_user_id`

服务端会按以下顺序处理：

1. 校验不能给自己创建私聊
2. 校验目标用户存在
3. 查询这两个用户之间是否已经有私聊
4. 若已存在则直接返回已有会话
5. 若不存在则新建 `sessions` 与两条 `session_members`

这样可以避免同一对用户在常规流程里反复创建新私聊，也为后续会话列表和历史消息查询保留稳定的 `session_id`。

### 2. 消息链路遵循“先落库，再推送”

当前文本消息发送通过 WebSocket 进入 `message service`，流程为：

1. 解析 `send_message`
2. 校验 `session_id`、文本非空、长度上限
3. 校验发送者是否属于该私聊
4. 插入 `messages`
5. 更新 `sessions.last_message_id / last_message_at`
6. 返回发送确认给发送方
7. 若接收方在线，则通过 `connection manager` 推送 `receive_message`

这条链路保持了 `docs/product/spec.md` 中“必须先落库，再推送”的约束，也与 `3.7 connection` 预留的单点推送能力自然衔接。

### 3. 3.8 当前只覆盖私聊，不提前展开群聊/历史消息

这次实现明确限制在 `3.8` 的范围内：

- 只支持 `private` 会话
- 只支持文本消息
- 只支持“发送即入库 + 在线单点推送”

暂未在本次中提前实现：

- 群聊发送
- 会话列表
- 历史消息查询
- 未读数统计

这样可以保证 3.8 交付的是一条最小、稳定、可验证的私聊主链路。

## 验证结果

- `cargo test` 通过
- `session service` 测试覆盖以下场景：
  - 创建新私聊成功
  - 已存在私聊时复用已有会话
  - 禁止自己和自己建私聊
- `session route` 测试覆盖以下场景：
  - 携带合法 token 可创建私聊
  - 缺少 token 被拒绝
- `message service` 测试覆盖以下场景：
  - 私聊成员发送文本消息成功
  - 非成员发送被拒绝
  - 空白文本被拒绝
- `connection handler` 最小链路测试覆盖以下场景：
  - 发送方收到 `message_sent`
  - 接收方在线时收到 `receive_message`
