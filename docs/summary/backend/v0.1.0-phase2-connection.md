## 关键实现说明

### 1. 握手阶段沿用现有 JWT 校验链路

`3.6 middleware` 已经完成 Bearer Token 的统一解析与 JWT 校验逻辑。

本次 WebSocket 握手没有直接复用 HTTP middleware，而是复用了同一套 JWT 服务与 Bearer 解析规则：

- 若握手请求携带 `Authorization` 头，则按现有 Bearer Token 规则解析
- 若客户端场景不方便携带请求头，则允许使用 `?token=<jwt>` 查询参数
- token 合法时才允许升级为 WebSocket 连接
- token 缺失或非法时直接返回 `401`

这样可以保证 HTTP 与 WebSocket 两条认证链路的 JWT 语义保持一致。

### 2. 当前连接管理采用“每个用户保留一个活动连接”

本次 connection manager 维护两层映射：

- `user_id -> 当前活动连接`
- `connection_id -> user_id`

当前策略为：

- 同一用户新建连接时，新的连接会覆盖旧的活动连接
- 旧连接后续断开时，不会误删新连接的在线状态

这个策略足够支撑当前 `3.7` 的“在线状态可管理、断线可清理”目标，也为后续 `3.8` 私聊单点推送提供了最小可用基础。

### 3. 基础协议先收口到连接确认与保活

为了避免在 `3.7` 提前进入消息业务协议，本次只定义最小基础事件：

- `connected`：握手成功后的首条服务端消息
- `ping`：客户端保活请求
- `pong`：服务端保活响应
- `error`：非法消息或当前阶段不支持的消息格式

这样后续在 `3.8 message + session` 中，可以在当前协议之上继续扩展发送消息、回执和推送事件，而不需要推翻连接层实现。

### 4. Postman 联调方式已收口

为了让 HTTP 登录和 WebSocket 握手可以串起来复用，本次将 Postman 调整为：

- HTTP collection 负责 `register / login / me`
- WebSocket collection 负责 `/ws` 握手验证
- `Login` 请求会自动把 JWT 写入 Postman 全局变量 `rustchat_token`
- WebSocket 请求直接引用 `{{rustchat_token}}`，避免两份 collection 间 token 不同步

这样联调顺序会比较稳定：

1. 先用 HTTP collection 登录
2. 自动拿到 `rustchat_token`
3. 再用 WebSocket collection 连接 `/ws`
4. 验证 `connected / ping / pong / error`

## 验证结果

- `cargo test` 通过
- `cargo check` 通过
- `/ws` 握手测试覆盖以下场景：
  - 合法 `Authorization` 头可升级连接
  - 合法查询参数 token 可升级连接
  - 缺少 token 返回 `401`
  - 非法 token 返回 `401`
- connection manager 测试覆盖以下场景：
  - 注册连接后用户在线状态可查询
  - 连接断开后映射可清理
  - 同一用户重复连接时，旧连接断开不会误删新连接状态
- Postman / 虚拟机手动联调已验证：
  - 登录后可通过导入的 WebSocket collection 成功连接 `ws://<vm-ip>:3000/ws?token=<jwt>`
  - 握手成功后服务端会先返回 `connected` 事件
  - 发送 `{"type":"ping"}` 可收到 `{"type":"pong"}`
  - 发送非法文本 `ping` 可收到 `{"type":"error","message":"invalid websocket message"}`
