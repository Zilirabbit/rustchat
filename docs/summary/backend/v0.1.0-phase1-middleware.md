## 关键实现说明

### 1. 鉴权职责从 extractor 收口到 middleware

此前 `CurrentUser` extractor 会自行：

- 读取 `Authorization` 头
- 解析 Bearer token
- 校验 JWT

这会让“鉴权逻辑”和“上下文读取逻辑”耦合在一起。

本次调整后：

- `middleware/auth.rs` 负责提取 Bearer token、校验 JWT、构造 `CurrentUser`
- `CurrentUser` extractor 只从请求上下文中读取已注入的用户信息

这样后续受保护接口只需要挂载中间件，即可复用同一套认证链路。

### 2. 受保护接口已形成完整 HTTP 认证链路

本次以 `GET /api/me` 为第一条正式接入 middleware 的受保护路由：

1. 请求进入路由级鉴权中间件
2. 中间件校验 Bearer token
3. 中间件将 `CurrentUser` 写入请求上下文
4. handler 通过 extractor 获取当前用户
5. 缺失或非法 token 统一返回 `401`

这与 `3.5 user` 阶段中“`JWT 校验 -> 当前登录用户提取 -> 受保护接口访问`”的联调提示保持一致，也为后续 `session / message / ws` 模块继续复用鉴权链路打好了基础。

### 3. 请求日志与错误日志能力已接入

新增请求日志中间件后，当前 HTTP 请求会记录：

- 请求方法
- 请求路径
- 响应状态码
- 请求耗时

同时，统一错误响应在输出时会补充日志：

- `4xx` 记录为 `warn`
- `5xx` 记录为 `error`

这让接口联调时可以同时从响应和服务端日志两侧定位问题。

### 4. 手动联调资料已补齐

考虑到当前阶段已经适合做真实 HTTP 联调，本次补充：

- `docs/api.md`
- `docs/postman/rustchat.postman_collection.json`

这样既能让开发者维护可读文档，也能直接导入 Postman 进行接口测试，并验证 middleware 在真实请求下的行为。
