## 关键实现说明

### 1. 会话列表查询优先复用现有表设计

本次没有额外新增表结构，而是直接复用 `3.4` 和 `3.8` 已经铺好的能力：

- `sessions.last_message_id`
- `sessions.last_message_at`
- `user_session_read_state.last_read_message_id`

这样会话列表可以直接拿到：

- 最近一条消息内容
- 最近消息时间
- 未读数量

也符合前面 summary 中“优先复用 `sessions.last_message_id / last_message_at`”的建议。

### 2. `private` 会话名称直接解析为“对方用户名”

当前 `GET /api/conversations` 返回的 `session_name` 规则为：

- 私聊：显示“当前用户之外的另一位成员用户名”
- 群聊：显示 `sessions.name`

这样前端在 v1 阶段不需要再额外请求一次“私聊对方资料”才能完成基础会话列表展示。

### 3. 未读数按“不是自己发送且超过已读游标”统计

当前未读数统计规则为：

- 只统计当前会话中的消息
- 不统计当前用户自己发送的消息
- 若存在 `last_read_message_id`，只统计 `id > last_read_message_id` 的消息
- 若当前用户在该会话下还没有 `user_session_read_state` 记录，则会把“他人发送的全部消息”视为未读

这与 `docs/spec.md`、`docs/db.md` 中“按当前用户 + 会话维度统计未读”的约束一致。

### 4. 本次只完成 4.2，不提前展开 4.3

这次实现明确限制在 `4.2 会话列表` 的边界内：

- 已完成：会话列表、最近消息、最近消息时间、未读数、排序
- 暂未实现：按会话分页查询历史消息、成员权限校验接口、已读状态写回接口

这样可以保证 `4.2` 先稳定闭环，再继续推进 `4.3 历史消息`。

## 验证方法

### 1. 基础变量配置

在 Postman 中确认：

- `base_url = http://<server-ip>:3000`
- `ws_base_url = <server-ip>:3000`

例如：

- `base_url = http://192.168.221.131:3000`
- `ws_base_url = 192.168.221.131:3000`

注意：

- `base_url` 不能只填 IP，必须包含 `http://` 和端口
- `ws_base_url` 按当前 collection 写法只填 `ip:port`，不要再手动加 `ws://`

### 2. 先验证 HTTP 服务可用

在 Postman 中依次执行：

1. `RustChat HTTP API -> System -> Health`
2. `RustChat HTTP API -> System -> DB Ping`

预期：

- `/health` 返回 `200`
- `/db/ping` 在数据库配置正常时返回 `200`

如果 `http://<server-ip>:3000/health` 无法访问，则不要继续做会话列表验证，应先检查：

- 服务是否已启动
- 服务是否监听 `0.0.0.0:3000`
- 防火墙或安全组是否放通 `3000`

### 3. 准备双用户数据

在 Postman 中按顺序执行：

1. `RustChat HTTP API -> Dual User Auth -> Register Alice`
2. `RustChat HTTP API -> Dual User Auth -> Register Bob`
3. `RustChat HTTP API -> Dual User Auth -> Login Alice`
4. `RustChat HTTP API -> Dual User Auth -> Login Bob`

说明：

- 若注册返回 `409 username already exists`，说明该用户已存在，可以继续登录
- 登录成功后，应自动写入以下全局变量：
  - `rustchat_alice_token`
  - `rustchat_bob_token`
  - `rustchat_alice_user_id`
  - `rustchat_bob_user_id`

如果后续 WebSocket 握手里看到：

- `ws://<ip>/ws?token=`
- 或报错连接到 `:80`

通常说明：

- `ws_base_url` 没带 `:3000`
- 或对应的 `rustchat_alice_token / rustchat_bob_token` 为空

### 4. 创建私聊会话

执行：

1. `RustChat HTTP API -> Session -> Create Private Session Alice -> Bob`

预期：

- 返回 `200`
- 响应中包含 `data.session_id`
- Postman 自动写入 `rustchat_private_session_id`

### 5. 通过 WebSocket 发送测试消息

使用真正的 WebSocket 请求页进行连接。

执行：

1. `RustChat WebSocket -> Private Chat -> Bob`，点击 `Connect`
2. `RustChat WebSocket -> Private Chat -> Alice`，点击 `Connect`

连接成功后都应先收到：

```json
{
  "type": "connected",
  "user_id": 1,
  "username": "alice",
  "connection_id": 1
}
```

然后发送测试消息：

先在 `Bob` 连接里发送两条：

```json
{"type":"send_message","session_id":{{rustchat_private_session_id}},"content":"bob-msg-1"}
```

```json
{"type":"send_message","session_id":{{rustchat_private_session_id}},"content":"bob-msg-2"}
```

再在 `Alice` 连接里发送一条：

```json
{"type":"send_message","session_id":{{rustchat_private_session_id}},"content":"alice-msg-1"}
```

预期：

- 发送方收到 `message_sent`
- 对方在线时收到 `receive_message`

### 6. 验证会话列表接口

当前 collection 中如果还没有现成的会话列表请求，可手动新增：

- 名称：`Get Conversations As Alice`
- 方法：`GET`
- URL：`{{base_url}}/api/conversations`
- Bearer Token：`{{rustchat_alice_token}}`

发送后预期：

- 返回 `200`
- 结果中包含 Alice 已加入的私聊会话
- `session_name = "bob"`
- `last_message = "alice-msg-1"`
- `last_message_time` 不为空
- `unread_count = 2`

原因：

- Bob 发了两条消息给 Alice
- Alice 自己发出的消息不会计入自己的未读数

再复制一份请求验证 Bob：

- 名称：`Get Conversations As Bob`
- Bearer Token：`{{rustchat_bob_token}}`

发送后预期：

- `session_name = "alice"`
- `last_message = "alice-msg-1"`
- `unread_count = 1`

原因：

- Alice 只给 Bob 发了一条
- Bob 自己发出的两条不会计入 Bob 的未读数

### 7. 验证排序规则

如果存在多个会话，可在某个较旧会话中再发送一条新消息，然后重新请求：

- `GET /api/conversations`

预期：

- 最近有新消息的会话排在前面
- 没有消息的会话排在最后

```json
{
  "code": 200,
  "message": "conversations fetched",
  "data": [
    {
      "session_id": 1,
      "session_type": "private",
      "session_name": "bob",
      "last_message": "alice-check-latest",
      "last_message_time": "2026-05-07 12:45:31.219061+00",
      "unread_count": 2
    }
  ]
}
```
