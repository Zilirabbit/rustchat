## 关键实现说明

### 1. 配置读取不再直接 panic

此前 `APP_PORT` 解析失败会直接触发 `expect`。本次改为由 `AppConfig::from_env()` 返回 `AppResult<AppConfig>`，统一纳入应用错误模型，便于后续继续扩展配置校验。

### 2. common 模块职责更清晰

当前 `common` 下已具备：

- `config.rs`：环境变量配置读取
- `logging.rs`：日志系统初始化
- `error.rs`：`AppError`、`AppResult`、接口错误响应转换
- `response.rs`：统一成功响应结构

这样后续 `user / session / message` 模块都可以直接复用，不需要各自维护一套响应和错误规范。

### 3. 接口响应格式统一

当前成功响应统一为：

```json
{
  "code": 200,
  "message": "service is healthy",
  "data": {
    "status": "ok"
  }
}
```

错误响应统一为：

```json
{
  "code": 503,
  "message": "database is not configured"
}
```

### 错误码扩展建议

当前 `code` 仍主要承载 HTTP 状态码语义，适合作为初始化阶段的统一返回，但在用户系统、鉴权、中间件、WebSocket 等模块接入后，建议进一步演进：

- 将 `HTTP status` 与 `business code` 拆开
- 前端与客户端优先依赖稳定的业务错误码，而不是仅依赖 `400 / 401 / 500`
- 业务错误码建议按模块分组命名，例如：
  - `COMMON_CONFIG_INVALID`
  - `AUTH_INVALID_TOKEN`
  - `AUTH_TOKEN_EXPIRED`
  - `USER_ALREADY_EXISTS`
  - `SESSION_FORBIDDEN`
- `AppError` 后续可统一提供：
  - `status_code()`
  - `error_code()`
  - `user_message()`
- 统一响应结构后续建议预留 `request_id` 字段，用于日志链路追踪与排障

该方案的目标是：

- 保持当前阶段实现简单
- 为后续模块增长保留稳定扩展点
- 让前后端、日志、监控都能围绕同一套错误语义协作
