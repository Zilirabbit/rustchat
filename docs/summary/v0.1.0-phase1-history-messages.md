## 关键实现说明

### 1. 历史消息继续放在 `message` 模块

本次没有新增单独的 history 模块，而是沿用项目结构约定，把历史消息查询放在 `message` 模块内：

- `message/dto.rs`：请求参数与响应结构
- `message/model.rs`：查询模型
- `message/repo.rs`：成员校验与数据库查询
- `message/service.rs`：参数校验、权限校验、分页组装
- `message/handler.rs` 与 `message/routes.rs`：HTTP 入口

这样可以和现有 WebSocket 发消息能力共用同一个消息业务边界。

### 2. 分页方式采用 `before_message_id`

接口格式为：

```text
GET /api/messages?session_id=<id>&limit=<1-50>&before_message_id=<optional>
```

返回顺序固定为：

```text
message_id DESC
```

也就是最新消息在前。继续加载更早消息时，使用上一页返回的 `next_before_message_id` 作为下一次请求的 `before_message_id`。

### 3. 成员权限校验复用 `session_members`

查询历史消息前会先检查：

```text
session_members(session_id, user_id)
```

只有当前登录用户属于该会话时才允许查询。非成员访问会返回 `403`，避免用户通过猜测 `session_id` 读取别人的聊天记录。

### 4. 查询路径复用既有索引

数据库查询条件为：

```text
WHERE m.session_id = $1
  AND ($2 IS NULL OR m.id < $2)
ORDER BY m.id DESC
LIMIT $3
```

这可以复用前面表设计中已经预留的：

```text
messages(session_id, id DESC)
```

本次不新增 migration。

### 5. 本次不提前实现已读写回

上一阶段 summary 提到后续可复用 `user_session_read_state` 补充已读进度更新。本次只完成 `4.3 历史消息` 的查询闭环，没有在拉取历史消息时自动写回已读状态。

原因：

- `4.3` 的任务范围是查询、分页、成员权限和返回结构
- 已读写回会改变 `4.2 会话列表` 的未读数行为，适合单独设计“进入会话 / 标记已读”接口

## 验证方法

### 1. 复用上一阶段基础验证

先按 `4.2 会话列表` summary 的验证方法准备环境：

1. 配置 Postman 变量：
   - `base_url = http://<server-ip>:3000`
   - `ws_base_url = <server-ip>:3000`
2. 验证：
   - `GET /health`
   - `GET /db/ping`
3. 准备双用户：
   - 注册 / 登录 Alice
   - 注册 / 登录 Bob
4. 创建私聊会话：
   - `POST /api/sessions/private`
   - 保存 `rustchat_private_session_id`
5. 通过 WebSocket 发送测试消息：
   - Bob 发送 `bob-msg-1`
   - Bob 发送 `bob-msg-2`
   - Alice 发送 `alice-msg-1`
6. 再验证上一阶段会话列表：
   - Alice 请求 `GET /api/conversations`
   - 预期 `session_name = "bob"`，`last_message = "alice-msg-1"`，`unread_count = 2`

### 2. 手动新增历史消息请求

当前 collection 中如果还没有历史消息请求，可手动新增：

- 名称：`Get History Messages As Alice`
- 方法：`GET`
- URL：`{{base_url}}/api/messages?session_id={{rustchat_private_session_id}}&limit=2`
- Bearer Token：`{{rustchat_alice_token}}`

发送后预期：

- 返回 `200`
- `message = "history messages fetched"`
- `data.session_id = {{rustchat_private_session_id}}`
- `data.limit = 2`
- `data.messages.length = 2`
- `data.messages[0].content = "alice-msg-1"`
- `data.messages[1].content = "bob-msg-2"`
- `data.has_more = true`
- `data.next_before_message_id` 不为空

原因：

- 历史消息按 `message_id DESC` 返回
- Alice 最后发送的 `alice-msg-1` 应排在第一页第一条

实际验证：

```json
{
  "code": 200,
  "message": "history messages fetched",
  "data": {
    "session_id": 1,
    "limit": 2,
    "before_message_id": null,
    "next_before_message_id": 7,
    "has_more": true,
    "messages": [
      {
        "message_id": 8,
        "session_id": 1,
        "sender_id": 1,
        "sender_username": "alice",
        "message_type": "text",
        "content": "alice-check-latest",
        "created_at": "2026-05-07 13:07:12.801074+00"
      },
      {
        "message_id": 7,
        "session_id": 1,
        "sender_id": 2,
        "sender_username": "bob",
        "message_type": "text",
        "content": "bob-msg-2",

  }
}
```

### 3. 验证游标分页

复制上一条请求，将 URL 改为：

```text
{{base_url}}/api/messages?session_id={{rustchat_private_session_id}}&limit=2&before_message_id=<上一步 data.next_before_message_id>
```

预期：

- 返回 `200`
- 返回更早的消息
- 如果只剩一条 `bob-msg-1`，则 `data.messages[0].content = "bob-msg-1"`
- 当没有更多旧消息时，`data.has_more = false`
- 当没有更多旧消息时，`data.next_before_message_id = null`

### 4. 验证 Bob 也可查询同一会话

新增或复制请求：

- 名称：`Get History Messages As Bob`
- 方法：`GET`
- URL：`{{base_url}}/api/messages?session_id={{rustchat_private_session_id}}&limit=20`
- Bearer Token：`{{rustchat_bob_token}}`

预期：

- 返回 `200`
- 能看到同一会话内的三条测试消息
- 顺序仍然是 `alice-msg-1`、`bob-msg-2`、`bob-msg-1`

### 5. 验证非成员不可查询

再注册 / 登录一个 Charlie，拿到 `rustchat_charlie_token` 后请求：

- URL：`{{base_url}}/api/messages?session_id={{rustchat_private_session_id}}&limit=20`
- Bearer Token：`{{rustchat_charlie_token}}`

预期：

### 6. 验证参数错误

分别请求：

```text
{{base_url}}/api/messages
```

```text
{{base_url}}/api/messages?session_id={{rustchat_private_session_id}}&limit=0
```

```text
{{base_url}}/api/messages?session_id={{rustchat_private_session_id}}&limit=51
```

预期：

- 缺少必填查询参数返回 `400`
- `limit = 0` 返回 `400`
- `limit = 51` 返回 `400`

验证结果：

```text

```

```json

  "code": 400,
  "message": "limit must be between 1 and 50"
}
```
